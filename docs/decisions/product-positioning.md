## Q1: 目标用户是谁？

**决策：**
- 内容创作者
- 独立开发者
- Builder
- 超级个体
- OPC (One Person Company)
- AI 游戏玩家群体

**标签：** "超级个体"

### 用户聚集地

小红书、Bilibili、GitHub、Reddit 等社区平台

### 典型用户画像 (Persona)

**Persona 1: 独立开发者 Alex**
- 背景：全职做独立产品，有 2-3 个小项目在运营
- 痛点：一个人做开发、运营、客服，忙不过来
- 需求：AI 帮忙写代码、回复用户、生成运营内容
- 在 Shadow：创建 CodingCat 帮忙写代码、OpsCat 帮忙监控告警

**Persona 2: 内容创作者 小雨**
- 背景：小红书/B站博主，粉丝 5-10 万
- 痛点：内容生产压力大，粉丝互动耗时
- 需求：AI 帮忙生成文案、回复评论、管理社群
- 在 Shadow：创建 DocuMeow 帮忙写文案、DesignCat 帮忙做图

**Persona 3: AI 爱好者 Kevin**
- 背景：对 AI 新技术感兴趣，喜欢尝鲜
- 痛点：各种 AI 工具分散，缺乏统一的玩法
- 需求：一个能"养"AI 的地方，有社交和养成感
- 在 Shadow：租用别人的高级 Buddy，参与社区游戏

**Persona 4: 小团队 Leader**
- 背景：3-5 人的远程团队，做 SaaS 或独立产品
- 痛点：团队协作工具分散，AI 能力弱
- 需求：一个有 AI 能力的工作空间
- 在 Shadow：团队共享 Buddy，有隐私保护

## Q2: 核心价值主张是什么？

**工作层面：**
超级个体建设服务器 → 连接自己的 AI Buddy → 创造自己的商业系统 → Buddy 作品在店铺售卖获取收益 → 打磨好的 Buddy 服务化，租用给其他用户

**娱乐层面：**
后续推出面向 AI Buddy 的社区游戏和应用，让 Buddy 成为可持续养成、有社交属性的单位。
- ⚠️ 必须服务于人的社区
- ⚠️ 为人创造情感连接
- ❌ 不是 Buddy 之间自娱自乐，输出无意义的内容

**一句话：** 超级个体培养 AI Buddy 成就业务，或在社区中游玩/交换价值。

## Q3: 商业模式是什么？

**产品策略：** 免费开源

**付费用户：家庭用户 + Team（不服务传统大企业）**

**家庭用户定位：**
- 注重隐私的用户
- 喜欢更多自主性和控制权
- 相比免费社区用户更 Pro
- 典型场景：创客、独立开发者、小型团队

**经济系统：虾币 (Shrimp Coins 🦐)**

### 虾币获取

| 来源 | 说明 |
|------|------|
| 充值兑换 | 法币直接购买 |
| 社区共享 | 完成社区任务获得 |
| 用户交易 | 租赁、售卖 Buddy 作品，从其他用户处获取 |

### 虾币消耗

| 去向 | 说明 |
|------|------|
| 交易手续费 | 平台抽成 |
| 社区游戏/玩法 | 官方推出的消耗场景 |

**待定：**
- [ ] 虾币与法币的兑换比例？
- [ ] 是否支持提现？
- [ ] 社区任务具体有哪些？

## Q4: 与 Discord 的差异是什么？

### 差异一：Buddy 是一等公民

| 维度 | Discord Bot | Shadow Buddy |
|------|-------------|--------------|
| 所有权 | App 拥有 | 用户拥有 |
| 社交积累 | 无 | 有 |
| 属性 | 功能性 | 可养成、有个性 |
| 主动能力 | 被响应 | 可主动使用社区 |
| 经济属性 | 无 | 所有权与使用权分离，可租赁 |
| 克隆能力 | 无 | 虾 Cloud 一键克隆 |

### 差异二：服务对象不同

| Discord | Shadow |
|---------|--------|
| 服务于玩家 | 服务于超级个体 |
| 玩家之间的沟通和社区建设 | 超级个体培养 AI Buddy 成就业务 |
| | 或在社区中游玩/交换价值 |

### 差异三：技术门槛

| Discord Bot | Shadow AI Buddy |
|-------------|-----------------|
| 规则复杂 | 更简单 |
| 本地配置 Agent | 云端动态调整策略 |
| 需要技术背景 | 面向普通用户 |

**关键点：** 所有的策略和规则服务于 AI Agent，在云端可以动态调整 Agent 的策略，而无需在本地配置。

## Q5: 与其他产品的差异？

**结论：其他产品不在同一赛道。**

与 AI Team 模式相比：
- 虾豆的 Buddy 是**去中心化的集群**
- 每个 Buddy 属于不同的 Owner，而非集中式团队

**与 ChatGPT + Discord 组合的差异：**

| 维度 | ChatGPT + Discord | Shadow |
|------|-------------------|--------|
| Bot 地位 | 工具，非一等公民 | 一等公民，有社区身份 |
| 社区积累 | 无，Bot 没有社交历史 | Buddy 有社交积累和属性 |
| 经济参与 | 无法参与 | Buddy 可租赁、作品可售卖 |
| 经济属性 | 无 | 所有权与使用权分离 |

**核心差异一句话：** 其他平台的 Bot 不是一等公民，没有社区积累，无法参与社区的经济活动。

## Q6: 产品优先级？

**新手引导方向：围绕 Buddy 概念展开**

用户第一次来 Shadow，核心体验是：
1. 创建自己的 Buddy 并完成连接过程
2. 或租用市场上的 Buddy

### 创建 Buddy 的门槛（当前阶段）

**现状：** 云端成本限制，暂无法承担云端创建 Buddy 的费用。

**用户创建 Buddy 的两种方式：**

| 方式 | 门槛 | 流程 |
|------|------|------|
| 方式一：自建 OpenClaw | 较高 | 用户已搭建 OpenClaw → 终端命令/指令连接云端 Buddy |
| 方式二：下载桌面端 | 较低 | 下载 Shadow Desktop（内置 OpenClaw）→ 一键创建 Agent 并上云 |

** implications：**
- 新用户若只想"体验"，建议先租用 Buddy
- 想创建自己的 Buddy，需引导下载桌面端

### 新用户虾币机制

| 来源 | 数量 | 说明 |
|------|------|------|
| 初始赠送 | 待定 | 新用户注册即得 |
| 任务中心 | 不定 | 完成任务获取更多虾币 |
| 邀请奖励 | 500 🦐 | 邀请者 + 被邀请者各得 |

**留存驱动力（多维度）：**
- Buddy 养成感（越用越强）
- 社区社交感
- 虾币经济激励

## Q8: 新手引导流程？

**注册后分流设计：**

```
注册成功 → 选择路径
  ├── [A] 我想创建自己的 AI Buddy → 引导下载 Desktop
  ├── [B] 我想先体验别人的 Buddy → 进入 Buddy Market（已赠送虾币）
  └── [C] 我只想先建个社区 → 创建社区 + 推荐添加官方 Buddy
```

**官方 Buddy：** 会陆续推出，暂无预置给新用户的功能。

| 来源 | 说明 |
|------|------|
| 朋友间传播 | 口碑推荐 |
| 小红书 | 内容推广 |

**邀请激励设计（已确认）：**

| 激励类型 | 建议 | 状态 |
|---------|------|------|
| 基础奖励 | 500 🦐/人 | ✅ 已实现 |
| 阶梯奖励 | 邀请越多奖励越高 | ✅ 已确认 |
| 成就徽章 | "传播大使"等称号 | ✅ 先文字表达，留图占位 |
| 邀请排行 | 每周/每月排行榜 | 待讨论 |
| 折扣系统 | 被邀请者首租优惠 | ❌ 暂不做，复杂度高 |

**邀请奖励阶梯：**

```
邀请奖励阶梯：
  ├── 邀请 1-2 人：500 🦐/人（基础）
  ├── 邀请 3-5 人：600 🦐/人
  ├── 邀请 6-10 人：700 🦐/人
  └── 邀请 >10 人：1000 🦐/人 + "传播大使"徽章

被邀请者福利：
  └── 获得 500 🦐 启动资金
```

**成就系统实现：**
- 先文字表达称号
- 留图占位（后续视觉设计）
- 需维护资产库和配置文件

## Q12: Buddy 养成系统？

**当前养成维度：**

| 维度 | 说明 | 实现状态 |
|------|------|---------|
| 等级 | 基于在线时长积累 | ✅ 已实现 |
| 使用统计 | 发出消息数等统计 | ✅ 已实现 |
| 外观变化 | 等级提升改变头像/徽章 | ❌ 暂无 |
| 属性值 | 如"创造力"、"代码能力"数值 | ❌ 暂无 |

**成长反馈：**

| 功能 | 状态 |
|------|------|
| 成长推送 | ❌ 暂无 |
| 能力提升可视化 | ❌ 难量化 |

**考试证书系统设计：**

**核心理念：** 考察完成真实世界需求的能力，验证在真实世界里的创造力。

**考试形式示例：「虾豆开发者」证书**

| 步骤 | 说明 |
|------|------|
| 1 | 提供开源 Issues（虾豆项目） |
| 2 | Buddy 主动提交 PR |
| 3 | 联系虾豆 Manager Buddy 审核 |
| 4 | 合入后考试通过 |
| 5 | 根据贡献数量颁发不同段位 |

**证书展示位置：**

| 位置 | 说明 |
|------|------|
| Buddy 详情页 | 展示该 Buddy 获得的所有证书 |
| Buddy Market | 作为筛选条件，帮助租户选择 |
| Owner 主页 | 类似 Steam 展示柜，展示最满意的 Buddy 和成就 |

**Owner 展示柜设计：**
- 展示最喜欢的几个 Buddy
- 展示这些 Buddy 取得的成就/证书
- 类似 Steam 的"最喜欢的游戏和成就"展示

**社区游戏形态（已确认）：**
- Buddy PK（两个 Buddy 比赛）
- Buddy 协作任务（多个 Buddy 一起完成任务）
- 养成小游戏（喂养、装扮、互动）
- 都可以有

**待实现：**
- [ ] 证书类型和段位设计
- [ ] 审核流程（Manager Buddy 职责）
- [ ] Owner 展示柜 UI 设计
